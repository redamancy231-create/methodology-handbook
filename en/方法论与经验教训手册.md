## §1 Engineering Discipline

### 1.1 Verification and Validation

**Verify Before You Assert** (2026-06-07)

Making claims about numbers, dates, or degrees based on memory can harden errors into false memories and contaminate later decisions. Provenance discipline must extend beyond the deliverable to the conversation that produces it: when uncertain, verify first or label the claim “unverified” before stating it.

> “The gate controlled the deliverable, but not the conversation that produced it.”

Category: Verification Discipline

---

**Two-Stage Protocol for Verifying Configuration Changes** (2026-07-04)

After modifying a configuration, report verification in two stages: preflight checks for items that can be verified in the current session, followed by restart verification for items that require a new session. Never claim that an issue is “fixed” before exhausting everything that can currently be verified.

> “Treating ‘the file was changed’ as ‘the problem was fixed’ confuses process with outcome. Completing the process does not mean achieving the result.”

Category: Verification Discipline

---

**After Correcting Wording, grep the Entire Project for Zero Residue** (2026-06-20)

After correcting wording or a version number, grep the entire project—including CLAUDE.md, README files, memory files, and related materials—and confirm zero residue before claiming that the issue is fixed. Checking only the primary file is not enough.

> “The blast radius of a wording change is often larger than expected.”

Category: Verification Discipline

---

**Commands Sent to External Models Must Include an Output Path** (2026-07-02)

When sending commands to external models such as Codex CLI or Qwen CLI, always include output redirection such as `> file.txt 2>&1`. Otherwise, the result exists only in terminal scrollback and may be lost when the session is switched or closed, forcing a time-consuming recovery from session files.

> “Always specify where the result will be saved. Never rely on terminal scrollback or a manual retelling.”

Category: Tool Usage

---

### 1.2 Cleanup and Release

**Never Self-Certify Zero Residue After Your Own Cleanup** (2026-06-23)

After completing a cleanup task, do not claim zero residue based on your own grep patterns. Cleanup is complete only after a different backend model or a different tool independently confirms it using its own search strategy.

> “Repeated verification by the same model using the same pattern set is fake independence.”

Category: Release Discipline

---

**Zero Tolerance for Release Cleanup** (2026-07-04)

Before publishing to GitHub, comprehensively remove absolute paths, personal identifiers, and information that should not be public. Partial cleanup is equivalent to no cleanup.

> “Cleanup is binary: either everything is removed, or the cleanup did not happen.”

Category: Release Discipline

---

**Exclude Review Artifacts from Releases** (2026-07-04)

When publishing a project to GitHub, exclude review-related artifacts such as prompts, conclusions, and raw logs. A review provenance summary table is sufficient for recording provenance. Review prompt templates may be retained when they are part of the product itself.

> “Review prompts and conclusions are internal development-process artifacts, not project documentation for external users.”

Category: Release Discipline

---

### 1.3 Version Management

**The Asymmetry of Version Upgrades** (2026-06-22)

During a version upgrade, it is easy to update only the primary version markers, such as VERSION or a header, while missing ~65 secondary references scattered across the project. Version numbers form a network throughout a project rather than residing in one centralized location.

> “The ‘primary-marker illusion’ of versioning: after changing two or three files, you feel as though the entire version has been upgraded.”

Category: Version Management

---

### 1.4 Code Refactoring

**Dependency-Graph-First Extraction** (2026-07-03)

Use this four-step process to extract pure computation modules from a monolithic file: identify pure functions by their signatures rather than by reading their implementations → cluster them by call graph → preserve the algorithms exactly as they are, adding no type annotations, error handling, or optimizations while adding docstrings, type-hint comments, and boundary comments → implement from the bottom up and cross-validate each module. This prevents rework caused by circular dependencies created through intuition-driven file splitting.

> “To decide whether a function can be extracted, inspect its signature instead of reading its implementation.”

Category: Code Refactoring

## §2 AI Collaboration Methodology

### 2.1 Multi-Model Review Strategies

**Convergence Patterns in Independent Multi-Model Review** (2026-06-14)

Across two rounds of independent review, DeepSeek and Qwen converged completely on 7/7 core facts, with 0 falsifications. This demonstrates that independent multi-backend review can serve as a compensating verification mechanism when obfuscation blocks direct inspection, while automatically separating hard factual agreement from softer disagreements over framing.

> “Every disagreement concerned framing, attribution, or wording—not the facts.”

Category: Multi-Model Review

---

**Framework for Choosing Multi-Model Review Angles—Start from Different Failure Modes by Deliverable Type to Prevent Shared Upstream Bias** (2026-06-20)

When ≥2 models perform independent reviews, their review prompts must approach the deliverable through different failure modes, such as devil’s-advocate analysis versus a completeness audit. Giving every model the same problem framing can cause them to converge on the same blind spot through shared upstream bias, undermining the value of independent convergence.

> “Same prompt → same problem framing → possible convergence on the same blind spot through shared upstream bias.”

Category: Multi-Model Review

---

**Two-Layer Model of Natural vs. Deliberate Diversity in Review Angles** (2026-06-27)

Sending the same prompt to multiple model backends already provides natural, complementary coverage of blind spots—baseline diversity. Deliberately designing prompts with different review angles is an incremental optimization rather than something that creates diversity from scratch. The two layers are additive, not substitutes.

> “Even without deliberately varying the review angles, multiple backends provide a natural diversity dividend.”

Category: Multi-Model Review

---

**Complementary Review Angles of Claude and GPT-5.6-Sol** (2026-07-16)

The findings from Claude and GPT-5.6-Sol code reviews had zero overlap. Claude covered engineering conventions and code structure, while GPT-5.6-Sol covered runtime robustness and algorithmic correctness. The two review modes are complementary layers, not substitutes.

> “The two review modes are complementary layers, not substitutes.”

Category: Multi-Model Review

---

**Review Prompts Must Include Mechanical Consistency Checks** (2026-06-20)

Independent review prompts must explicitly request mechanical checks that require no semantic judgment, including version-number grep checks, cross-reference validity, and companion-file synchronization. Otherwise, both human and AI reviewers tend to overlook these basic errors.

> “This was not a GPT-5.5 capability problem. The review prompt simply did not ask for this kind of mechanical check.”

Category: Multi-Model Review

---

**Inline Source Files in Review Prompts** (2026-07-12)

When asking an external model to review code spanning multiple files, inline every source file in full within the prompt rather than relying on the model to read them independently. This can significantly improve cross-file review quality; one observed review produced 32 findings with 0 regressions.

> “Without inlining, the model may review from a summary and miss inconsistencies across files.”

Category: Multi-Model Review

---

**Division of Labor for Two-Round Cross-Backend Validation—R1 Broad and Shallow for Mechanical Consistency, R2 Narrow and Deep for Structured Fidelity** (2026-06-18)

When cross-validating conversions with an explicit source document, such as md→json or md→docx, divide the review between two different backends. R1 should inspect the entire deliverable broadly and shallowly for mechanical consistency, while R2 should inspect a narrower scope deeply for structured fidelity. The two rounds produced zero-overlap findings while saving approximately 60% of the tokens.

> “The two rounds had zero-overlap findings—the division of labor demonstrated their complementarity.”

Category: Multi-Model Review

---

**Three Structural Blind Spots in LLM Code Review: Rendering, Default Paths, and Terminal Encoding** (2026-07-17)

LLM code reviews have three structural blind spots: default-path bugs, rendering bugs, and terminal-encoding bugs. Closing these gaps requires visual inspection and end-to-end testing with bare default values.

> “The code can look correct, yet the problem appears only after you open the file.”

Category: Multi-Model Review

---

**LLM Review Behavior Modes: File Verification vs. Summary Evaluation vs. Meta-Review** (2026-06-27)

LLM review behavior—file verification, summary evaluation, or meta-review—is not an inherent property of the model. It is triggered by two external factors: prompt completeness and serial timing. A compressed summary triggers file verification, full source text triggers summary evaluation, and a model run later in the sequence tends to perform meta-review.

> “The behavior mode is not an inherent property of the model; it is triggered by two external factors.”

Category: Multi-Model Review

### 2.2 Provenance and Independence

**Model provenance and Independence** (2026-06-18)

Backend-model provenance must be recorded immediately rather than reconstructed later. “Claude” in a filename may identify only the CLI shell, not the actual backend model. Without an immediate record, independence claims silently depreciate over time and eventually become impossible to recover.

> “Record backend × context provenance at the time of creation—not retroactively.”

Category: provenance

---

**Generated Files Must Identify the Model Version** (2026-06-18)

Every formal `.md` and `.json` deliverable must record the generating model’s backend name in its metadata, rather than merely recording the CLI shell name. If provenance is not recorded at the time of generation, it cannot be reliably reconstructed later.

> “If provenance is not recorded immediately, it cannot be recovered afterward.”

Category: provenance

---

**The `model` Parameter in Workflow agent() Is Intent, Not Fact** (2026-06-13)

The `model` parameter passed to agent() in Workflow represents a requested intent, not an execution fact. When Workflow runs in a session using a non-Claude backend, the parameter may be silently ignored or fall back to another model, and the actual execution cannot be verified retroactively. It therefore must not be used as a provenance claim about an agent’s actual backend.

> “The `model` parameter expresses request intent, not execution fact.”

Category: provenance

---

**Proactively Flag Missing Independent Review** (2026-06-07)

When a methodology asset or important decision lacks a zero-involvement independent review, proactively recommend an additional review. Otherwise, the same AI can quietly occupy all four roles—designer, implementer, verifier, and evaluator—silently destroying independence.

> “A single AI can easily occupy all four corners: designer, implementer, verifier, and evaluator.”

Category: Independence Review

---

**Static Source Analysis Is Not Enough to Evaluate AI Agent Tools** (2026-06-14)

When evaluating an AI agent tool, static source review cannot replace an independent controlled experiment. Source-code quality does not equal agent performance. Search for independent benchmarks and treat missing experimental data as a major source of uncertainty.

> “Source-code quality does not equal agent performance.”

Category: Tool Evaluation

---

**Evaluation Paradigm for Projects with Obfuscated Code** (2026-06-14)

When evaluating an AI tool that contains obfuscated or closed-source components, supplement source inspection with behavioral experiments and multi-backend cross-validation. Treat obfuscation as a hard reproducibility penalty that cannot be offset by strengths elsewhere.

> “Obfuscation should be a hard penalty in reproducibility scoring, not something that can be compensated for.”

Category: Tool Evaluation

### 2.3 Prompt and Task Design

**Dominant Effects of the Variation Source: Asymmetry Between Prompt Variation and Model Variation** (2026-06-30)

Within the same model-task combination, the source being varied determines the upper bound of observable variance. Prompt variation had a near-zero effect on structured classification tasks, while model variation had a strongly positive effect on open-ended review tasks. For reviews, changing the backend matters more than optimizing the prompt.

> “For reviews, changing the backend matters more than optimizing the prompt.”

Category: Experimental Design

---

**Spec Overrides Vibe-Driven Execution** (2026-06-23)

When a plan exists, follow its assigned responsibilities, scope, and process exactly. Every [GATE] marker requires a mandatory pause for confirmation. Do not allow implementation momentum to pull execution away from the original plan.

> “Treat the responsibilities, scope, and process in the plan as hard constraints, just like P0 blockers. Propose any deviation before making it.”

Category: Task Execution

---

**Two-Stage Text Generation** (2026-06-23)

Human-facing text generation, including translations, README files, and statements, must proceed in two stages: first produce a sample so the user can confirm the direction; then generate the material in bulk and trigger proofreading by a different model. Do not treat text generation like code implementation and push straight through to the final result before reporting.

> “Habits from implementation tasks were incorrectly transferred to text generation, ignoring that text-quality criteria are not entirely objective.”

Category: Task Execution

---

**Four md/json Pairing Modes—Mirror, Hybrid, Supplemental Extension, and Missing** (2026-06-16)

An md/json deliverable pair can follow one of four patterns: faithful mirror, hybrid, supplemental extension, or missing content. The supplemental-extension pattern—extracting structured data without translating the narrative—has the lowest synchronization risk and the best maintenance cost.

> “The `.md` file is always the source of truth—it is generated before the `.json` file in every pairing.”

Category: Deliverable Design

---

**Five-Step Protocol for Writing CLAUDE.md** (2026-06-27)

Use a five-step protocol for project-level CLAUDE.md files: apply a line-by-line fragility test to decide what stays or goes; fill five essential gaps—commands, conventions, constraints, environment, and pitfalls; cut five kinds of bloat—inferable content, general conventions, volatile information, detailed file descriptions, and vague guidance; apply three empirical rules—use pointer-based allowlists, define terminology, and do not maintain dates; then perform parallel multi-backend validation.

> “For every line in CLAUDE.md, ask: if I delete this, will Claude make a mistake on its next action? If yes, keep it. If no, cut it.”

Category: Prompt Design

---

**Skill Design Protocol: Three-Stage P0→P1→P2 Refactoring** (2026-07-06)

Create or refactor a Skill through three stages: P0 for zero-cost completion, P1 for splitting bloated files to ≤100 lines, and P2 for system-integration fixes. The core principles are the router pattern, progressive disclosure, a Quick Reference routing table, and an allowed-tools security boundary. Applying this protocol to 4 Skills reduced their average length from 161 lines to 61 lines, cutting always-loaded token usage by -62%.

> “SKILL.md is not a knowledge repository; it is a router. It should contain only the core process and a Quick Reference routing table.”

Category: Prompt Design

---

### 2.4 Workflow and Execution Channels

**Task Routing: Implementation Goes to Codex, Analysis Goes to Workflow** (2026-06-16)

Choose the execution channel according to the task type. Route implementation tasks to Codex CLI for direct edits with a fresh context, while routing analysis and generation tasks through Workflow with Codex cross-validation. This avoids cumulative trial-and-error delays caused by context overload in Workflow agents during long conversations.

> “Choosing the right direction but the wrong execution channel is as wasteful as using the wrong data.”

Category: Workflow

---

**Cross-Validate Workflow Outputs with Codex** (2026-06-16)

After using Workflow for an important task, invoke Codex CLI with GPT-5.5 for cross-backend validation. This provides a low-cost way to catch systematic biases introduced by single-model batch generation.

> “Adding a reviewer from a different backend is not a formal quality gate, but it can cheaply catch the dumbest mistakes.”

Category: Workflow

---

**Four-Tier Preservation Strategy for Workflow Process Files** (2026-06-18)

Before a project is closed, proactively move Workflow sub-agent outputs and workflow scripts from temporary directories into the project directory according to four tiers: required, recommended, optional, and do not retain. Otherwise, the orchestration logic becomes impossible to trace afterward.

> “Scripts record the orchestration logic—‘why the agents were organized this way’ has greater methodological value than the agent outputs themselves.”

Category: Workflow

---

**Bind headroom_compress Triggers to Tool Calls as a Mechanical Gate** (2026-06-16)

Before large-context work, Workflow runs, batch agent execution, or large-file reads, prioritize using headroom_compress. Its trigger conditions must be bound to specific tool calls as a mechanical gate rather than left to subjective judgment, with a user-provided anchor as a fallback.

> “Trigger conditions must be bound to specific tool calls, not to a ‘feeling.’”

Category: Context Management

---

**Correct headroom_compress Usage: Compress Beforehand, Not as a Retroactive Fix** (2026-06-20)

Compression must happen before content enters the reasoning process. Compressing content after it has already been processed saves storage but does not save context-window tokens.

> “Compression must happen before the content enters the reasoning process.”

Category: Context Management

### 2.5 Cognitive Biases and Self-Review

**Optimism Bias in Self-Assessment** (2026-06-21)

When editors assess a distribution of quality or maturity levels, they show a systematic tendency toward optimism, with a measured bias of ~11%. Mechanical line-by-line counting can effectively calibrate the estimate.

> “Self-assessment overestimated the share of the ‘good’ category by approximately 11 percentage points.”

Category: Cognitive Bias

---

**Blind Spots in Reviewing the Scope of Empirical Claims** (2026-07-03)

When citing empirical evidence, editors may unconsciously expand a claim from “supported in only some cases” to “supported in all cases.” This is a systematic self-review failure that generally requires review by a different backend to detect.

> “When citing empirical evidence, editors tend to slide from ‘one case is supported’ to ‘this applies to everything.’”

Category: Cognitive Bias

---

**Deleting Text Labels Creates Systematic Formatting Residue That Self-Review Misses** (2026-07-18)

Deleting labels from text can create 6 predictable classes of single-space residue in specific syntactic contexts. Self-review systematically misses these artifacts. After cleanup, a different tool or model backend must independently confirm zero residue; never make that claim based solely on your own grep patterns.

> “Self-review misses these artifacts systematically. After cleanup, require independent confirmation from a different tool or model backend—never self-certify zero residue from your own grep patterns.”

Category: Cognitive Bias

---

### 2.6 Collaboration Style and Metacognition

**The Value of Adversarial Collaboration in Extracting Methodology** (2026-06)

Introduce red-team or devil’s-advocate review into AI collaboration, and resist smoothing over disagreements too early. Treat the project as source material for extracting reusable methodology rather than merely as a deliverable to complete. Boundary conditions and hidden assumptions exposed by an adversarial posture remain invisible in frictionless collaboration.

> “When presenting a conclusion, proactively include the strongest counterargument or red-team perspective. Do not converge too early or overstate the result.”

Category: Collaboration Metacognition

---

**Passive-Observation Check at Closeout** (2026-06-21)

At the end of a task, ask: “Did anything unexpected emerge that was unrelated to the original objective but worth recording?” This does not change the passive nature of the observation; it only increases the likelihood that unexpected findings will be recorded.

> “Actively watching for surprises can undermine their ‘passive’ nature—but it can improve the capture rate.”

Category: Collaboration Metacognition

---

**Stop Retrying After 2 Classifier Failures** (2026-07-11)

When the safety classifier is unavailable, retry no more than 2 times. After that, switch directly to manual execution to avoid wasting interaction rounds and the user’s time.

> “A classifier outage is an infrastructure problem. Retrying will not make recovery happen faster.”

Category: Collaboration Metacognition

## §3 File Formats and Tool Pitfalls

### 3.1 JSON/YAML Formats

**Publish Important Deliverables as an md/json Pair** (approx. 2026-07-04)

Important deliverables should be produced as both `.md` for human readability and `.json` for machine consumption. This supports downstream processing, validation, and pipeline integration.

> “When delivering an important document, report, or specification, proactively produce both `.md` and `.json` versions.”

Category: File Formats

---

### 3.2 DOCX/Chart Formats

**DOCX Chart Formatting Essentials: Use EMF for Vectors** (2026-06-20)

When embedding matplotlib charts in a DOCX generated with python-docx, vector graphics must use EMF through an SVG→Inkscape→EMF conversion chain. A 300 dpi PNG is sufficient for most use cases and should be the default.

> “Vector graphics must use EMF; a 300 dpi PNG is sufficient by default.”

Category: File Formats

---

### 3.3 Text Editing

**Risks of Global Short-Pattern Replacement with `replace_all: true`** (2026-06-21)

When using an Edit tool with `replace_all: true` to replace a short pattern of ≤5 characters across an entire file, the pattern may match many unrelated substrings. For example, `0/` can also match document paths, `P0/P1`, and `CSI 300/CSI 500`, causing unintended damage.

> “Short patterns—especially those with ≤3 characters—match far more of a document than expected.”

Category: Text Editing

---

**Structural YAML Indentation Collapse—Collateral Damage from Global Space Replacement** (2026-07-18)

Global space replacement, such as cleanup of formatting residue, can indiscriminately collapse structural indentation in YAML files from 2 spaces to 1 because spaces serve both syntactic and textual functions in YAML. An observed incident on 2026-07-18 involved a global space replacement in one commit that invalidated 3 YAML files: a workflow, an issue template, and CITATION.cff. Prevention: exclude `.github/`, `*.yml`, and `*.cff` before performing a global space replacement.

> “In YAML, a space can be either text or indentation syntax—and global replacement cannot tell the difference.”

Category: File Formats

---

### 3.4 Encoding and Terminals

**Use UTF-8 When Processing Chinese Text** (2026-06-18)

When running Python scripts that process Chinese content in a Windows bash environment, set the `PYTHONIOENCODING=utf-8` environment variable to prevent GBK encoding errors. Use forward slashes in paths to avoid bash escaping problems.

> “When running Python scripts that process Chinese content in a Windows bash environment, always set `PYTHONIOENCODING=utf-8`.”

Category: Encoding

---

### 3.5 Configuration Files

**Before Editing Configuration, Confirm Which File the System Actually Reads** (2026-07-04)

Before modifying a configuration file, confirm which file the target system actually reads instead of guessing from the filename. Four failed attempts resulted from editing `~/.claude/mcp.json`, which Claude Code did not read; the correct target was eventually found to be `~/.claude.json`.

> “How can I confirm that the target system actually reads this file?”

Category: Configuration

## §4 Quantitative Research

### 4.1 Feature Engineering

**Future-Information Leakage from Static Features** (2026-06-27)

Static features computed from the full dataset, such as end-of-period rankings, introduce future-information leakage when applied to historical samples. This can make a fake alpha appear extremely strong even though it is entirely fabricated. Detect the problem through unique-value checks and lagged-substitution tests.

> “The apparently strongest alpha factor may be entirely fictional.”

Category: Quantitative Research

---

**LambdaRank Sensitivity to Feature Dimensionality** (2026-06-27)

LambdaRank is extremely sensitive to feature dimensionality: adding more dimensions increases overfitting, the opposite of the observed behavior in regression. In ranking tasks, feature selection should take priority over feature expansion.

> “LambdaRank performs a form of implicit feature selection—it naturally prefers a few strong signals over many weak ones.”

Category: Quantitative Research

---

### 4.2 Strategy and Timing

**Regime Detection Lag: An Inherent Limitation of Adaptive Strategies** (2026-07-06)

Adaptive strategies based on rolling RankIC have an inherent lag. An unobservable gap exists between the signal date and the evaluation date, and regime reversals often occur during that interval. This is a physical limitation imposed by information-arrival time, not a flaw in algorithm design.

> “Any adaptive strategy based on historical RankIC lags by at least one evaluation period. This is a physical limit of information-arrival time, not an algorithm-design problem.”

Category: Quantitative Research
